Cloudflare Pages는 무료로 사용할 수 있는 최고의 호스팅 서비스 중 하나입니다. 대역폭 무제한, 빠른 글로벌 CDN, 자동 SSL까지 제공합니다. 하지만 무료 플랜에는 한 가지 중요한 제한이 있습니다. 바로 월 500회 빌드 제한입니다.

500회라고 하면 많아 보이지만, 바이브코딩을 하다 보면 작은 수정 하나에도 푸시를 하게 됩니다. "버튼 색깔 바꿔줘" → 푸시 → "아 더 진하게" → 푸시. 이런 식으로 하루에 수십 번 배포하다 보면 금방 제한에 도달합니다. 효율적인 배포 전략을 세워야 합니다.

---

## 월 500회 빌드 현실적인 계산

![gXV00UQy.webp](https://pub-bc48937f7a09402fbd548e79d043ffa0.r2.dev/images/cloudflare-muryoui-hangye-bildeu/7b096c716d18a6c2b34ad43bd02da505b321dca4.webp)

월 500회를 30일로 나누면 하루 약 16회입니다. 하루 16번이면 넉넉해 보이지만, 실제로 개발 중에는 쉽게 넘길 수 있는 숫자입니다. 특히 바이브코딩 초반에는 AI에게 이것저것 요청하고 결과를 확인하느라 배포 횟수가 많아집니다.

한 달 동안 여러 프로젝트를 진행하면 상황이 더 빡빡해집니다. Cloudflare 계정 전체에서 500회가 공유되기 때문입니다. 프로젝트 3개를 운영하면 각각 월 160회 정도만 사용할 수 있습니다. 계획적인 배포가 필요합니다.

---

## 하루 1회 배포 전략

가장 효과적인 방법은 로컬에서 충분히 테스트하고 하루에 한 번만 배포하는 것입니다. VSCode에서 Live Server 확장을 사용하면 브라우저에서 실시간으로 결과를 확인할 수 있습니다. 푸시하지 않아도 로컬에서 수정 결과를 볼 수 있으니 배포 횟수를 크게 줄일 수 있습니다. 꼭 1회 배포할 필요는 없습니다. 문서를 수정하고 삭제하고 하는 과정에서 한 번 수정, 삭제 할 때마다 모두 1회 배포로 취급합니다. 더군다나 클라우드 플레어 측의 오류로 디플로이가 안됐을 경우 역시 1회 배포로 취급하기도 합니다. 그래서 5개 10개의 블로그 포스팅을 한 번에 배포하는 지혜가 필요합니다.

하루 동안 작업한 내용을 저녁에 한 번에 커밋하고 푸시하는 습관을 들이세요. 급한 수정이 아니라면 다음 날로 미루는 것도 방법입니다. 월말에 빌드 횟수가 부족해지면 배포를 못 하는 상황이 생길 수 있으니 여유를 두고 관리하세요.

---

## 브랜치 전략 활용하기

개발용 브랜치와 배포용 브랜치를 분리하는 것도 좋은 방법입니다. 평소에는 dev 브랜치에서 작업하고 커밋합니다. dev 브랜치 푸시는 Cloudflare 배포를 트리거하지 않도록 설정합니다. 변경 사항이 충분히 쌓이면 main 브랜치에 병합(merge)해서 한 번에 배포합니다.

Cloudflare Pages 설정에서 Production branch를 main으로 지정하고, Preview branch는 비활성화하거나 특정 브랜치만 지정할 수 있습니다. 이렇게 하면 개발 중 잦은 푸시는 빌드에 영향을 주지 않고, 실제 배포할 때만 빌드 횟수가 소모됩니다.

## 빌드 횟수 모니터링

현재 빌드 사용량은 Cloudflare 대시보드에서 확인할 수 있습니다. Workers & Pages 메뉴에서 프로젝트를 선택하면 이번 달 빌드 횟수가 표시됩니다. 정기적으로 확인하면서 페이스 조절을 하세요.

월초에 빌드를 많이 소진했다면 월말에는 배포를 자제해야 합니다. 반대로 월말인데 빌드가 많이 남았다면 평소에 미뤄뒀던 수정 사항들을 한꺼번에 배포해도 됩니다. 매달 1일에 빌드 횟수가 리셋되니 월말에 남은 빌드는 아껴둘 필요가 없습니다. 클라우드 플래어에서 제공하는 api를 사용해서 빌드횟수가 얼마나 되는지 첵크할 수 있습니다. 아래는 바이브 코딩을 통해 만든 블로그 관리기. 상단에 빌드 현황이 나오게 만들었습니다.

## 제한이 부족하다면

월 500회로 정말 부족하다면 몇 가지 대안이 있습니다. 첫째, Cloudflare Pro 플랜(월 20달러)으로 업그레이드하면 월 5,000회 빌드가 제공됩니다. 본격적으로 사이트를 운영한다면 고려해 볼 만합니다.

둘째, 여러 호스팅 서비스를 병행하는 방법도 있습니다. 메인 프로젝트는 Cloudflare Pages, 실험용 프로젝트는 Vercel이나 GitHub Pages를 사용하면 제한을 분산할 수 있습니다. 셋째, 로컬 빌드 후 결과물만 배포하면 빌드 시간을 줄이고 효율을 높일 수 있습니다.

## 결론

Cloudflare Pages 무료 플랜의 월 500회 빌드 제한은 계획적으로 사용하면 충분한 양입니다. 로컬에서 테스트하고 하루 1회 배포하는 습관, 브랜치 전략 활용, 정기적인 사용량 모니터링으로 효율적으로 관리하세요. 무료 서비스의 한계를 이해하고 현명하게 활용하는 것도 개발 능력의 일부입니다. 다음 강에서는 GitHub Actions 무료 사용량에 대해 알아보겠습니다.
